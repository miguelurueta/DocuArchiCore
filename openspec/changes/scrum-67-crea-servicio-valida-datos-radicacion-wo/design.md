## Problem Statement

PROMPT ARQUITECTÓNICO — Servicio de validación de campos para construcción de tarea workflow Rol esperado: Arquitecto de software senior y desarrollador backend .NET Nombre del Servcio: ValidaDatosRadicacionTareaWorkflowService OBJETIVO Implementar un servicio interno de validación encargado de revisar los datos de radicación antes de la construcción o registro de la tarea de workflow, con el fin de evitar errores funcionales o técnicos por incumplimiento de restricciones de campos. El servicio debe validar, como mínimo: longitud máxima de campos (maxlength) formato esperado del campo valores obligatorios compatibilidad básica del dato con el destino funcional La validación debe ejecutarse antes de registrar la tarea de workflow, para evitar errores posteriores en el flujo. ALCANCE FUNCIONAL El servicio únicamente valida datos en memoria. No debe: consultar base de datos usar repositorios leer tablas depender de QueryOptions depender de DapperCrudEngine registrar datos modificar persistencia El servicio debe tomar la estructura de RelacionCamposRutaWorklflow y validar que los valores asignados en DatoCampoPlantilla cumplan las reglas funcionales mínimas requeridas para que la construcción o registro de la tarea workflow no falle. PARÁMETROS DE ENTRADA List<RelacionCamposRutaWorklflow> relacionCamposRutaWorklflow Si el proyecto ya cuenta con un DTO específico de validación o configuración de restricciones, reutilizarlo. No crear DTOs innecesarios. RETORNO Definir un contrato explícito y consistente. Opción sugerida: debe adaptarse al reterno de la funcion ValidaCamposRadicacionAsync Task<AppResponses<>> Donde el resultado permita conocer: si la validación fue exitosa qué campos fallaron qué regla falló qué mensaje técnico o funcional debe devolverse UBICACIÓN DE COMPONENTES Si la ruta no existe en el repositorio crear la ruta. Servicio Proyecto: MiApp.Services Ruta: /Service/Radicacion/RelacionCamposRutaWorklflow Modelo o DTO de soporte Proyecto: MiApp.Models o MiApp.DTOs Ruta: /Radicacion/Models/RelacionCamposRutaWorklflow Documentación Repositorio: DocuArchiCore Ruta: /Docs/Radicacion/RelacionCamposRutaWorklflow ARQUITECTURA APLICABLE Al tratarse de un servicio interno de validación, debe seguir el flujo: Service ↓ DTO / Model ↓ Validación en memoria REGLA ARQUITECTÓNICA IMPORTANTE No generar capas innecesarias. No crear Repository No crear Controller No crear acceso a base de datos No crear lógica de persistencia La implementación debe mantenerse mínima, clara y consistente con la arquitectura del proyecto. OBJETIVO FUNCIONAL DETALLADO El servicio debe recorrer la estructura RelacionCamposRutaWorklflow y validar cada DatoCampoPlantilla de acuerdo con las reglas funcionales definidas por cada campo. Como mínimo debe soportar validaciones de: Longitud máxima si el valor supera la longitud permitida, registrar error controlado Formato texto numérico fecha correo electrónico identificador otros formatos definidos por la estructura real del proyecto Requerido si el campo es obligatorio y no tiene valor, registrar error Nulos y vacíos diferenciar null, string vacío y espacios en blanco cuando aplique Compatibilidad básica evitar enviar al workflow datos que funcionalmente no puedan ser procesados REQUERIMIENTOS FUNCIONALES Implementar patrón Service. La respuesta debe estar envuelta en AppResponses<T>. La función debe estar envuelta en bloque try/catch. El servicio debe validar todos los campos de entrada y retornar el detalle de errores encontrados. La validación debe basarse únicamente en los datos disponibles en memoria. Si no existen errores de validación: success = true data = lista vacía o resultado sin errores message = "Validación exitosa" Si existen errores funcionales de validación: success = false o success = true con detalle de errores, según el estándar actual del proyecto retornar el detalle completo de errores encontrados En caso de excepción: success = false message descriptivo errors o ErrorMessage = ex.Message según el estándar actual del proyecto No inventar reglas de validación no definidas por la estructura real del proyecto. Si un campo no tiene suficiente metadata para validar, documentar esa limitación sin romper el flujo completo. REQUERIMIENTOS TÉCNICOS Crear interfaz del servicio en el mismo archivo que la clase si esa es la convención del proyecto. Registrar interfaz y clase en Program.cs. Documentar con comentarios XML: propósito de la clase propósito del método descripción de parámetros valor de retorno comportamiento cuando no hay errores comportamiento cuando hay errores de validación comportamiento ante excepciones Mantener nombres consistentes con el estándar del proyecto. Aplicar nomenclatura PascalCase en clases, métodos, propiedades, DTOs e interfaces. Mantener la lógica de validación centralizada en el Service. No mezclar lógica de validación en DTOs o Models. Si existen validadores reutilizables en el proyecto, reutilizarlos antes de crear lógica nueva. PRINCIPIOS ARQUITECTÓNICOS MÍNIMOS Single Responsibility El servicio debe encargarse únicamente de validar los campos requeridos para el workflow. Separation of Concerns La validación debe estar separada de la construcción de datos y del registro de la tarea. Dependency Inversion Si se usan validadores auxiliares, deben inyectarse por interfaz. Testability El servicio debe ser totalmente testeable sin infraestructura externa. Minimal Architecture No crear capas ni componentes que no aporten valor real al flujo. Reutilización Reutilizar modelos, DTOs o estructuras de error existentes antes de crear nuevos artefactos. REGLA DE FIDELIDAD FUNCIONAL La validación debe respetar exactamente las restricciones funcionales necesarias para evitar errores en la construcción o registro de la tarea workflow. No: inventar formatos inventar longitudes máximas inventar campos obligatorios alterar reglas de negocio sin justificación Si una regla no está claramente definida en la estructura disponible, documentarla como pendiente o aplicar únicamente validaciones seguras y confirmadas.
