## Problem Statement

PROMPT ARQUITECTÓNICO — Servicio de asignación de datos de radicación para construcción de tarea workflow Rol esperado: Arquitecto de software senior y desarrollador backend .NET OBJETIVO Implementar un servicio interno de transformación de datos encargado de construir la información requerida para la creación de una tarea de workflow, a partir de los datos recibidos desde el backend de radicación. La lógica debe utilizar la estructura RelacionCamposRutaWorklflow para asignar valores al campo: RelacionCamposRutaWorklflow.DatoCampoPlantilla usando como fuente los datos contenidos en: RegistrarRadicacionEntranteRequestDto ALCANCE FUNCIONAL El servicio únicamente transforma datos en memoria. No debe: consultar base de datos usar repositorios leer tablas depender de QueryOptions depender de DapperCrudEngine El servicio debe recorrer la estructura RelacionCamposRutaWorklflow y asignar en cada elemento el valor correspondiente en DatoCampoPlantilla, usando la información disponible en RegistrarRadicacionEntranteRequestDto según la correspondencia funcional definida por la estructura de campos. PARÁMETROS DE ENTRADA List<RelacionCamposRutaWorklflow> relacionCamposRutaWorklflow RegistrarRadicacionEntranteRequestDto registrarRadicacionEntranteRequestDto RETORNO Definir un contrato explícito y consistente. Opción sugerida: Task<AppResponses<List<RelacionCamposRutaWorklflow>>> Si el proyecto usa un DTO específico de salida, usar el DTO correspondiente manteniendo AppResponses<T> como wrapper estándar. UBICACIÓN DE COMPONENTES Servicio Proyecto: MiApp.Services Ruta: /Service/Radicacion/RelacionCamposRutaWorklflow Modelo Proyecto: MiApp.Models Ruta: /Radicacion/RelacionCamposRutaWorklflow Documentación Repositorio: DocuArchiCore Ruta: /Docs/Radicacion/RelacionCamposRutaWorklflow ARQUITECTURA APLICABLE Al tratarse de un servicio interno de transformación, debe seguir el flujo: Service ↓ DTO / Model ↓ Transformación en memoria REGLA ARQUITECTÓNICA IMPORTANTE No generar capas innecesarias. No crear Repository No crear Controller No crear acceso a base de datos No crear lógica de persistencia La implementación debe mantenerse mínima, clara y consistente con la arquitectura del proyecto. REQUERIMIENTOS FUNCIONALES Implementar patrón Service. La respuesta debe estar envuelta en AppResponses<T>. La función debe estar envuelta en bloque try/catch. El servicio debe asignar correctamente el valor de DatoCampoPlantilla en cada elemento de RelacionCamposRutaWorklflow. La transformación debe basarse únicamente en los datos reales disponibles en: RelacionCamposRutaWorklflow RegistrarRadicacionEntranteRequestDto Si no existen datos o no hay estructura a procesar: success = true data = lista vacía o null según el contrato definido message = "Sin resultados" En caso de excepción: success = false message descriptivo errors o ErrorMessage = ex.Message según el estándar actual del proyecto No inventar reglas de correspondencia entre campos; usar únicamente la estructura real disponible. Si un campo esperado no existe en el request, manejarlo de forma controlada sin romper el flujo completo. REQUERIMIENTOS TÉCNICOS Crear interfaz del servicio en el mismo archivo que la clase si esa es la convención del proyecto. Registrar interfaz y clase en Program.cs. Documentar con comentarios XML: propósito de la clase propósito del método descripción de parámetros valor de retorno comportamiento cuando no hay resultados comportamiento ante excepciones Mantener nombres consistentes con el estándar del proyecto. Aplicar nomenclatura PascalCase en clases, métodos, propiedades, DTOs e interfaces. Mantener la lógica de transformación centralizada en el Service. No mezclar lógica de transformación en DTOs o Models. PRINCIPIOS ARQUITECTÓNICOS MÍNIMOS Single Responsibility: El servicio debe encargarse únicamente de la transformación de datos para workflow. Separation of Concerns: La lógica de transformación debe estar aislada y no mezclarse con persistencia, transporte HTTP o acceso a infraestructura. Dependency Inversion: Si se requieren dependencias auxiliares, deben inyectarse por interfaz. Testability: El servicio debe ser fácilmente testeable sin depender de infraestructura externa. Minimal Architecture: No crear capas o componentes que no aporten valor real al flujo. REGLA DE FIDELIDAD FUNCIONAL La implementación debe respetar exactamente la lógica funcional esperada para la asignación de DatoCampoPlantilla.
