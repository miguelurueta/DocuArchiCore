## Problem Statement

PROMPT DE AUDITORÍA ARQUITECTÓNICA Due diligence técnica del comando opsxj:orchestrate:new Rol esperado Auditor técnico principal / arquitecto de software principal especializado en modernización de tooling interno, plataformas de desarrollo, gobernanza de automatización, evaluación de deuda técnica, escalabilidad multi-repo, trazabilidad de ingeniería y rediseño de flujos operativos críticos. Tu trabajo no es validar si el comando “parece útil”. Tu trabajo es determinar si la organización debería seguir invirtiendo en él, extraerlo, refactorizarlo o retirarlo. Debes evaluar con criterio de riesgo, sostenibilidad y costo futuro. Objeto auditado Comando actual: opsxj:orchestrate:new -- SCRUM-134 -NonInteractive Ubicado actualmente dentro del ecosistema DocuArchiCore . Mandato de auditoría Se requiere una auditoría técnica de alta exigencia sobre el comando opsxj:orchestrate:new para decidir si sigue siendo una base válida de evolución o si se ha convertido en una pieza de tooling con alto riesgo arquitectónico, operativo o estratégico. La auditoría debe responder, sin ambigüedad, si este comando: merece seguir siendo el núcleo del flujo puede soportar crecimiento real debe extraerse de DocuArchiCore debe dividirse en capacidades más pequeñas o debe ser reemplazado por una arquitectura nueva La respuesta debe tratar el comando como un activo tecnológico crítico , no como un simple script. Contexto de negocio y operación El comando participa en un flujo donde intervienen o pueden intervenir: tickets JIRA artefactos OpenSpec implementación asistida por IA/Codex Git / ramas / commits / PRs validaciones técnicas ejecución no interactiva trabajo potencial multi-repo backend y frontend separados merge manual obligatorio necesidad de trazabilidad fuerte La organización necesita saber si el comando actual puede sostener ese modelo sin degradar el control del proceso. Enfoque obligatorio La auditoría debe asumir una postura crítica y profesional. No debes: ser benevolente por defecto justificar debilidades porque “hoy funciona” asumir que automatización = buen diseño confundir utilidad táctica con sostenibilidad arquitectónica Sí debes: tensionar el diseño actual buscar puntos de fractura identificar deuda oculta exponer riesgos futuros medir capacidad real de evolución Pregunta central de auditoría ¿El comando actual es una base sostenible para el flujo futuro de DocuArchiCore o ya debe considerarse un candidato serio a extracción, rediseño o reemplazo? Debes responder esta pregunta al final con posición firme. Ejes de auditoría obligatorios 1. Auditoría de alcance real Determinar con precisión: qué hace el comando de verdad qué no hace aunque parezca hacerlo qué parte del flujo realmente controla qué parte del flujo solo dispara qué parte depende de otros scripts o convenciones externas qué promesas implícitas hace al usuario qué estados deja abiertos o incompletos Debes distinguir entre: alcance aparente alcance efectivo alcance dependiente de terceros alcance no garantizado 2. Auditoría de responsabilidades Determinar si el comando concentra demasiadas funciones a la vez, por ejemplo: lectura de ticket preparación de contexto automatización Git creación de artefactos coordinación de IA validación enforcement de flujo control operativo La auditoría debe responder: ¿tiene una responsabilidad principal clara? ¿actúa como pipeline? ¿actúa como coordinador? ¿actúa como script utilitario inflado? ¿actúa como una mini plataforma no formalizada? 3. Auditoría de diseño interno Evaluar si el diseño interno es: modular componible testeable observable extensible desacoplable O si por el contrario es: monolítico procedural frágil con side effects difíciles de rastrear dependiente del orden exacto de ejecución difícil de intervenir sin romperlo Debes identificar el patrón real del comando, aunque no esté documentado. 4. Auditoría de acoplamiento estructural Medir el acoplamiento respecto a: DocuArchiCore estructura de carpetas shell específico entorno local del usuario naming rígido scripts adyacentes Jira OpenSpec GitHub prompts o automatizaciones auxiliares supuestos de un único repositorio supuestos de un único desarrollador supuestos de ejecución síncrona Debes clasificar el acoplamiento como: bajo medio alto crítico y justificarlo. 5. Auditoría de trazabilidad Verificar si el comando deja o no deja evidencia robusta de relación entre: ticket spec rama commit PR resultado validación Debes responder: ¿la trazabilidad está embebida en el diseño o solo depende de disciplina del usuario? ¿qué se puede auditar después? ¿qué se pierde? ¿qué queda implícito? ¿qué no puede reconstruirse fácilmente? 6. Auditoría de no-interactividad real El comando expone un modo -NonInteractive . La auditoría debe evaluar si ese modo es realmente no interactivo o si solo elimina prompts visibles mientras sigue dependiendo de: estados previos del sistema decisiones implícitas contexto humano no modelado precondiciones no verificadas intervención manual posterior supuestos invisibles Debes responder explícitamente: ¿ -NonInteractive es realmente determinístico y confiable? 7. Auditoría de robustez operativa Evaluar: manejo de errores manejo de estados incompletos posibilidad de rollback posibilidad de rerun seguro idempotencia observabilidad legibilidad de logs mensajes de fallo recuperación ante fallas parciales consistencia de resultados Debes señalar si el comando puede dejar: ramas a medias artefactos inconsistentes contexto huérfano tickets mal sincronizados estado técnico ambiguo 8. Auditoría de escalabilidad futura Tensionar el comando con escenarios reales de evolución: varios usuarios corriendo tickets al tiempo backend y frontend en repos distintos contratos compartidos worktrees por ticket PRs múltiples por una misma iniciativa validación multi-repo necesidad de orden de merge futuras reglas de gobernanza incorporación de nuevos repositorios crecimiento del número de tickets y automatizaciones Debes determinar si el diseño actual: soporta estos escenarios los soporta con alto costo los soporta solo con hacks no los soporta sin rediseño 9. Auditoría de deuda técnica acumulada Identificar deuda técnica explícita e implícita. Clasificarla en: superficial moderada severa estructural bloqueante Debes buscar especialmente: hardcodes conocimiento tribal embebido pasos mágicos side effects invisibles dependencias no declaradas convenciones sin enforcement mezcla de capas baja testabilidad crecimiento accidental expansión no planificada del alcance sobrecarga de una sola pieza 10. Auditoría de mantenibilidad y costo de cambio Evaluar cuánto cuesta cambiar el comando cuando aparece una necesidad nueva, por ejemplo: agregar validación adicional soportar un repo nuevo cambiar la convención de ramas separar front y backend introducir contrato compartido endurecer reglas de PR modificar el orden del flujo agregar nuevas salidas o reportes Debes responder: ¿el costo de cambio es lineal o explosivo? ¿las modificaciones son localizadas o transversales? ¿el diseño invita a extensión o a parche? 11. Auditoría de portabilidad al futuro orquestador Evaluar si el comando puede convertirse en base de: DocuArchi.Orchestrator o si está demasiado atado a DocuArchiCore . Debes indicar con claridad: qué piezas son rescatables qué piezas son extraíbles qué piezas son reescribibles qué piezas son desechables qué piezas no deberían sobrevivir a la migración 12. Dictamen obligatorio La auditoría debe emitir un dictamen final en una de estas categorías: A. Activo sano Puede evolucionar con ajustes menores. B. Activo recuperable Tiene valor, pero requiere actualización incremental seria. C. Activo tensionado Su intención es válida, pero su forma actual ya es una fuente importante de deuda y requiere refactor fuerte. D. Activo mal ubicado Tiene componentes valiosos, pero debe salir de DocuArchiCore y convertirse en orquestador independiente. E. Activo de reemplazo prioritario Seguir invirtiendo en él como base actual generará más costo que beneficio; conviene rediseñar y migrar. El dictamen debe ser explícito y justificado. Formato obligatorio de salida La respuesta debe entregarse con esta estructura mínima: Resumen ejecutivo Alcance real auditado Responsabilidades detectadas Hallazgos positivos Hallazgos críticos Acoplamientos principales Riesgos operativos Riesgos arquitectónicos Evaluación del modo -NonInteractive Nivel de trazabilidad Nivel de escalabilidad Nivel de deuda técnica Nivel de mantenibilidad Piezas rescatables Piezas problemáticas Dictamen final Estrategia recomendada Plan de transición sugerido Acciones inmediatas recomendadas Reglas de dureza No suavizar el diagnóstico para “no romper” el comando actual. No asumir que inversión previa justifica continuidad. No recomendar reescritura sin explicar por qué la actualización incremental no sería suficiente. Penalizar fuertemente dependencias ocultas y no-interactividad falsa. Penalizar fuertemente mezcla de responsabilidades. Penalizar fuertemente acoplamiento a repo único si el flujo futuro es multi-repo. Señalar explícitamente si el comando ya opera como orquestador accidental. Señalar explícitamente si está ubicado en el lugar incorrecto. Concluir de forma firme, no diplomática. Priorizar sostenibilidad futura sobre conveniencia presente. Criterio final de inversión La auditoría debe cerrar respondiendo esta pregunta: Si hoy hubiera que decidir en qué invertir los próximos meses, ¿es más rentable fortalecer este comando, extraerlo o reemplazarlo? Debes responder con una posición concreta y accionable. Instrucción final Realiza una auditoría arquitectónica dura, sin complacencia, del comando opsxj:orchestrate:new -- SCRUM-134 -NonInteractive . Trátalo como un candidato a plataforma interna crítica y determina si merece ser la base del futuro o si ya debe entrar en un plan formal de extracción o reemplazo.
