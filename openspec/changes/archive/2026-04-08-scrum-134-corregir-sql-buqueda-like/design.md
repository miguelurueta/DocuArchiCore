## Problem Statement

PROMPT ARQUITECTONICO - Ticket 02 BE Corregir sintaxis SQL de busqueda LIKE en Workflow Inbox para motores multi-engine Rol esperado: Arquitecto de software senior backend (.NET, SQL, seguridad, query builders dinamicos, compatibilidad multi-engine). OBJETIVO Corregir la generacion de condiciones LIKE usadas por Workflow Inbox para eliminar la sintaxis invalida causada por el uso actual de ESCAPE '\\\\' . La solucion debe: Garantizar compatibilidad de sintaxis con MySQL/MariaDB. Garantizar compatibilidad de sintaxis con Oracle. Garantizar compatibilidad de sintaxis con SQL Server. Preservar el comportamiento funcional existente. Mantener seguridad ante caracteres especiales en patrones LIKE . Evitar regresiones en: rows count export autocomplete CONTEXTO EXISTENTE Componente afectado: MiApp.Services/Service/Workflow/BandejaCorrespondencia/WorkflowInboxQueryBuilder.cs Funcionalidad relacionada: SearchType = 2 -> busqueda global LIKE . ApplyLikeSearch(search, dynamicColumns) . BuildAutocomplete(...) . WorkflowInboxQueryPolicy.EscapeLikeLiteral(...) . WorkflowInboxQueryPolicy.LikeEscapeClause . Tests backend: DocuArchiCore/tests/TramiteDiasVencimiento.Tests/WorkflowInboxQueryBuilderTests.cs Nota sobre el estado actual: El builder actual construye condiciones RawConditions como literales SQL escapados. No usa parametros para el patron LIKE . No introducir parametrizacion nueva en este ticket salvo que el alcance se amplie formalmente para modificar QueryOptions y DapperCrudEngine . UBICACION ESPERADA Modificar unicamente si aplica: MiApp.Services/Service/Workflow/BandejaCorrespondencia/WorkflowInboxQueryBuilder.cs Actualizar tests backend en: DocuArchiCore/tests/TramiteDiasVencimiento.Tests/WorkflowInboxQueryBuilderTests.cs No modificar DapperCrudEngine , QueryOptions , DTOs publicos, controller, endpoint ni frontend salvo aprobacion explicita del alcance. RESTRICCIONES OBLIGATORIAS No modificar el endpoint POST /api/workflowInboxgestion/inboxgestion . No modificar AppInputSearch . No modificar frontend para ocultar el error. No cambiar contrato publico del backend. No introducir SQL dinamico inseguro. No romper SearchType = 1 . No romper SearchType = 3 . No alterar la semantica de filtros estructurados fuera del alcance de este ticket. No tocar DapperCrudEngine salvo necesidad formalmente aprobada. REGLAS DE IMPLEMENTACION Reemplazar el uso actual de ESCAPE '\\\\' en Workflow Inbox. Usar un caracter de escape explicito, estable y compatible como literal SQL: ! La clausula esperada para el comportamiento comun debe ser: ESCAPE '!' Ajustar WorkflowInboxQueryPolicy.LikeEscapeClause para que no emita ESCAPE '\\\\' . Ajustar EscapeLikeLiteral(...) para escapar correctamente: El propio caracter de escape elegido: ! -> !! % -> !% _ -> !_ comillas simples mediante la logica existente de EscapeSqlLiteral . Evitar doble escape inconsistente. La funcion debe tener una unica responsabilidad: normalizar el literal LIKE y luego proteger comillas simples. No asumir que [ y ] tienen la misma semantica en todos los motores: En SQL Server pueden tener significado especial en patrones LIKE . En MySQL/MariaDB y Oracle no deben forzar una sintaxis incompatible. Si se requiere tratamiento de [ y ] , debe hacerse de forma documentada por motor o dejarse como literal comun si la estrategia multi-engine lo justifica. Mantener la generacion actual de literales escapados en RawConditions , sin introducir parametros nuevos en este ticket. Aplicar la misma logica de escape en: ApplyLikeSearch(...) BuildAutocomplete(...) Mantener consistencia funcional con: SearchType = 1 -> sin LIKE global. SearchType = 2 -> LIKE global. SearchType = 3 -> busqueda avanzada sanitizada. No generar condiciones LIKE si el termino normalizado es null , vacio o whitespace, ni generar patrones inutiles como LIKE '%%' . REGLAS DE SEGURIDAD Evitar SQL injection por texto de busqueda. Evitar SQL injection por patrones LIKE . Escapar comillas simples con EscapeSqlLiteral . Escapar % , _ y el caracter de escape elegido de forma deterministica. Asegurar que runtime y pruebas usen la misma funcion de escape. No permitir bypass mediante combinaciones como !% , !! , O'Hara , 100% , A_B . REGLAS DE CONSISTENCIA La misma condicion LIKE debe aplicarse en: consulta de filas ( Build ) consulta de conteo ( BuildCount ) exportacion ( BuildExport ) autocomplete ( BuildAutocomplete ) No se permite divergencia entre rows, count y export. Autocomplete puede usar patron de prefijo, pero debe usar la misma estrategia de escape y la misma clausula ESCAPE . COMPATIBILIDAD MULTI-ENGINE OBLIGATORIA El cambio debe producir SQL sintacticamente valido para: MySQL MariaDB Oracle SQL Server Estrategia requerida: Usar la clausula SQL estandar ESCAPE '!' para todos los motores anteriores. Generar patrones LIKE donde ! sea el unico caracter de escape. No emitir ESCAPE '\\\\' bajo ningun caso. No depender de reglas de backslash propias de MySQL. No depender de reglas de corchetes propias de SQL Server. No depender de sintaxis especifica de un unico motor. Comportamiento esperado por motor: Motor Sintaxis requerida Ejemplo de patron valido MySQL LIKE '%100!%%' ESCAPE '!' 100% como texto literal MariaDB LIKE '%100!%%' ESCAPE '!' 100% como texto literal Oracle LIKE '%100!%%' ESCAPE '!' 100% como texto literal SQL Server LIKE '%100!%%' ESCAPE '!' 100% como texto literal Para SQL Server, si se requiere que [ y ] se traten como literales de forma especial, la solucion no debe hacerlo con una regla que rompa MySQL/MariaDB/Oracle. Debe documentarse como una variante por motor o dejarse fuera del alcance de este ticket si el builder no recibe provider/motor. Si durante la implementacion se demuestra que un motor requiere una variante no compatible con la estrategia comun ESCAPE '!' , se debe detener el cambio y documentar la decision arquitectonica antes de modificar QueryOptions , DapperCrudEngine o agregar deteccion de provider. PRUEBAS UNITARIAS OBLIGATORIAS Actualizar WorkflowInboxQueryBuilderTests para cubrir: Escape correcto de % : 100% -> 100!% . Escape correcto de _ : A_B -> A!_B . Escape correcto del caracter de escape elegido: A!B -> A!!B . Escape de comilla simple: O'Hara . Trim aplicado correctamente. Search vacio, null o whitespace no genera condicion LIKE . Search demasiado corto no genera condicion LIKE , respetando LikeSearchMinLength . El SQL generado no incluye ESCAPE '\\\\' . El SQL generado incluye ESCAPE '!' . El SQL generado usa una forma valida para MySQL/MariaDB/Oracle/SQL Server. Build , BuildCount y BuildExport mantienen la misma condicion LIKE . BuildAutocomplete usa el mismo escape y no contiene ESCAPE '\\\\' . SearchType = 1 no agrega LIKE . SearchType = 3 no cambia por este ticket. Casos minimos de busqueda: 100% A_B A!B O'Hara texto con espacios alrededor PRUEBAS DE INTEGRACION / CALIDAD Validar que la busqueda con caracteres especiales no retorna error 400.
