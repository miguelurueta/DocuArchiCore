## Problem Statement

PROMPT ARQUITECTÓNICO — MIGRACIÓN .NET csfc_verifica_existencia_radicado Rol esperado: Arquitecto de software senior y desarrollador backend .NET especializado en migración de sistemas legacy. OBJETIVO Migrar la función legacy csfc_verifica_existencia_radicado hacia la arquitectura actual del proyecto DocuArchiCore, preservando completamente el comportamiento funcional original. Crea la nueva funcion SolicitaExistenciaRadicadoRutaWorkflow La migración debe respetar: - Arquitectura por capas - Principios SOLID - Seguridad en consultas - Uso de DapperCrudEngine - Uso de QueryOptions - Wrapper estándar AppResponses - Alias de conexión definidos en el proyecto - Fidelidad funcional respecto al código legacy La migración debe basarse exclusivamente en el código legacy real, extrayendo: - firma funcional - queries reales - reglas de negocio - dependencias de contexto - estructura de salida - tablas afectadas - dependencias externas RUTA DESTINO REPOSITORIO OBIGATORIO Ruta Repositorio destino : MiApp.Repository/Workflow/RutaTrabajo Ruta DTO destino : MiApp.DTOs/DTOs/Workflow/RutaTrabajo Ruta Model destino : MiApp.Models/Models/Workflow/RutaTrabajo Ruata Documentación : Docs/Workflow/RutaTrabajo ARQUITECTURA OBLIGATORIA Si la migración solo requiere acceso a datos → implementar solo Repository. Si existe lógica de negocio → implementar Service + Repository. Si se expone vía API → implementar Controller + Service + Repository. Reglas: - Controller no accede directamente al Repository - Repository no contiene lógica de negocio - Service contiene la lógica de negocio - DTO transporta datos - Model representa entidades PRINCIPIOS ARQUITECTÓNICOS La implementación debe respetar SOLID: Single Responsibility Cada clase debe tener una única responsabilidad. Open / Closed El código debe poder extenderse sin modificar comportamientos existentes. Liskov Substitution Las implementaciones deben poder sustituir sus interfaces. Interface Segregation Las interfaces deben ser pequeñas y específicas. Dependency Inversion Las dependencias deben abstraerse mediante interfaces e inyección de dependencias. PRINCIPIO DE FIDELIDAD FUNCIONAL La migración debe replicar exactamente el comportamiento del código legacy. Orden de migración: 1. Analizar función legacy 2. Extraer reglas de negocio 3. Replicar comportamiento 4. Validar funcionalidad 5. Optimizar arquitectura si aplica Nunca modificar reglas de negocio durante la migración. IDENTIFICACIÓN DE BASE DE DATOS LEGACY Conectores posibles: Dbase_Conction_Mysql -> workflowtconta Dbase_Conction_Mysql_RA -> docuarchi Dbase_Conction_Mysql_DA -> docuarchi Alias en entorno actual: workflowtconta -> MySqlConnection_WF docuarchi -> MySqlConnection_DA La migración debe detectar el conector antes de construir la consulta. Todas la funciones deben incluir el paramtro defaultDbAlias REGLA DE MIGRACIÓN DETERMINISTA La migración no debe inventar lógica ni inferir estructuras. Se deben extraer del código legacy: - tablas - joins - filtros - ordenamientos - cálculos - validaciones Si algún comportamiento no puede determinarse, documentarlo como pendiente. MIGRACIÓN DE CONSULTAS Proceso: 1. localizar query legacy 2. identificar tablas 3. identificar joins 4. identificar filtros 5. identificar ordenamiento 6. migrar usando QueryOptions Nunca concatenar SQL manualmente. SEGURIDAD Todas las consultas deben: - ser parametrizadas - evitar SQL Injection - usar QueryOptions RESPUESTA ESTÁNDAR Todas las funciones deben retornar: Task<AppResponses> Si existen múltiples registros: Task<AppResponses<List>> MANEJO DE RESULTADOS Si no existen registros: success = true data = [] message = “Sin resultados” MANEJO DE ERRORES Toda función debe estar envuelta en try/catch. Los errores deben retornar un AppResponses controlado. PRINCIPIO DE ATOMICIDAD Si la función realiza múltiples operaciones: - Insert - Update - Delete Debe utilizar transacciones. Todas las operaciones deben completarse o ninguna. PRINCIPIO DE REUTILIZACIÓN Antes de crear nuevos DTO, Model, Repository o Query, verificar si existe una implementación reutilizable en el proyecto. Evitar duplicación de lógica. PRINCIPIO DE COMPATIBILIDAD DE CONTRATOS La migración no debe romper contratos existentes. Esto incluye: - APIs existentes - DTOs consumidos por frontend - servicios internos PRINCIPIO DE OBSERVABILIDAD Las funciones migradas deben incluir logs controlados y trazabilidad del flujo. PRINCIPIO DE PERFORMANCE Evitar: - SELECT * - consultas innecesarias - N+1 queries Seleccionar únicamente columnas necesarias. REGLA SOBRE SESSION LEGACY Nunca utilizar: Session.Item HttpContext.Current.Session FindControl HiddenFields Si aparecen dependencias legacy: documentar variable explicar su uso proponer posibles reemplazos solicitar decisión arquitectónica DOCUMENTACIÓN Ruta: Docs/Workflow/RutaTrabajo Archivos: SCRUM-[ID]-Diagramas.md Debe incluir: diagrama de casos de uso diagrama de clases diagrama de secuencia diagrama de estados si aplica Si la migración implementa API SCRUM-[ID]-Integracion-Frontend.md descripción técnica del DTO de respuesta parámetros requeridos para invocar el servicio o endpoint origen de los parámetros ejemplo de request ejemplo de response dirección de la API consideraciones de consumo desde frontend SCRUM-[ID]-Migracion-Legacy.md PRUEBAS Unit Tests: - respuesta exitosa - sin resultados - excepción controlada Integration Tests: - MySQL - Testcontainers - Docker ENTREGABLES Repository migrado Service si aplica DTO si aplica Model si aplica Controller si aplica Unit tests Integration tests Documentación técnica Documento de trazabilidad de migración RESTRICCIONES No: - romper arquitectura existente - concatenar SQL - inventar queries - inventar DTOs - modificar reglas de negocio - usar Session legacy La migración debe basarse únicamente en el código legacy real y la estructura de la base de datos correspondiente.
