## Problem Statement

PROMPT ARQUITECTÓNICO Completar contrato backend de menuItems con soporte de Children e IsDivider para AppDropdown Rol esperado: Arquitecto de software senior y desarrollador backend .NET OBJETIVO Extender el contrato backend de acciones dinámicas para permitir que la columna de acciones en workflowInboxgestion renderice y ejecute correctamente menús  contextuales en frontend, alineado con AppDropdown. El contrato debe soportar: resolución completa de acciones sin hardcodeo en frontend submenús mediante Children divisores mediante IsDivider ARQUITECTURA A RESPETAR Backend -> DynamicUiTableDto -> Frontend dinámico -> AppDropdown Reglas: no hardcodear catálogo en frontend mantener compatibilidad hacia atrás del contrato actual mantener UiActionDto como contrato base mantener compatibilidad con tablas dinámicas actuales SOLUCIÓN DEFINIDA (OBLIGATORIA) DynamicUiTableDto debe extenderse con: public List<UiActionDto> MenuActions { get; set; } = []; ESTRUCTURA DEL CONTRATO CellActions mantiene el trigger BehaviorConfig.menuItems es: List<string> de ActionId MenuActions catálogo de acciones completas solo incluye acciones relevantes para la tabla actual soporta jerarquía mediante Children soporta divisores mediante IsDivider EXTENSIÓN DE UiActionDto Se debe extender con: public List<UiActionDto>? Children { get; set; } public bool IsDivider { get; set; } REGLAS DE CONTRATO toda acción ejecutable debe tener ActionId toda acción ejecutable debe tener ActionId único en toda la respuesta, incluyendo Children los divisores quedan excluidos de la regla de unicidad menuItems debe ser exclusivamente List<string> de ids no permitir mezcla de ids y objetos en menuItems REGLAS DE RESOLUCIÓN Frontend debe: leer menuItems resolver contra MenuActions construir AppDropdownItem[] Reglas: resolución por igualdad exacta de ActionId si MenuActions es null o viene vacío, frontend no debe romper render si un id no existe, frontend debe ignorar el item sin error fatal debe existir fallback controlado cuando falte resolución REGLAS DE SUBMENÚS (Children) Children representa estructura jerárquica directa Children NO utiliza menuItems Children ya viene completamente resuelto desde backend la resolución es recursiva cada child sigue contrato UiActionDto REGLAS DE DIVISORES Si IsDivider = true: no debe tener ActionId no debe tener Behavior no debe tener Request no debe tener Children frontend lo renderiza como separador visual no es ejecutable VALIDACIONES ESTRUCTURALES no permitir ActionId duplicados entre acciones ejecutables no permitir divisor con Children no permitir divisor con Behavior no permitir estructuras recursivas inválidas no permitir referencias a acciones inexistentes en menuItems RESPONSABILIDAD BACKEND construir MenuActions completo garantizar unicidad de ActionId en acciones ejecutables incluir metadata de ejecución incluir Rules y Claims por acción resolver jerarquía mediante Children incluir divisores con IsDivider no depender de lógica frontend RESPONSABILIDAD FRONTEND resolver ids contra MenuActions mapear a AppDropdownItem[] mapear Children a children mapear IsDivider a separator visual evaluar Rules y Claims ejecutar acciones reutilizando la action layer existente EJEMPLOS VÁLIDOS Caso válido simple: BehaviorConfig.menuItems = ["reasignar_tramite", "archivar_tramite"] MenuActions contiene acciones con ActionId = "reasignar_tramite" y ActionId = "archivar_tramite" Caso válido con submenú: una acción de MenuActions incluye Children cada child viene completamente definido como UiActionDto Caso válido con divisor: un item de MenuActions tiene IsDivider = true no tiene ActionId, Behavior, Request ni Children EJEMPLOS INVÁLIDOS menuItems mezcla strings y objetos dos acciones ejecutables comparten el mismo ActionId un divisor tiene Behavior un divisor tiene Children menuItems referencia ids inexistentes sin catálogo resoluble en MenuActions COMPATIBILIDAD si MenuActions no viene o viene vacío, frontend no debe romper render si menuItems referencia ids inexistentes, esos items se ignoran la solución no debe afectar otras acciones dinámicas ya existentes fuera del menú contextual CRITERIOS DE ACEPTACIÓN menuItems se resuelve completamente contra MenuActions el dropdown funciona sin hardcodeo en frontend los submenús funcionan correctamente los divisores se renderizan correctamente cada acción ejecutable se resuelve y ejecuta correctamente existe fallback controlado si falta resolución el contrato queda reusable para cualquier tabla dinámica RESTRICCIONES no romper la arquitectura actual no romper compatibilidad del contrato público existente no introducir lógica en controller no acoplar la solución a un módulo específico
