TICKET JIRA — BE (Listo para copiar y pegar) Título: Corregir GetInitialContentAsync: hidratar plantilla usando ra_editor_template_tokens (required/optional) y fuentes system/static/editable/respuesta_api Tipo: Bug / Mejora técnica Componente: Editor (Plantillas / Initial Content) Prioridad: Alta Bloquea: Generación de htmlInicial (plantilla) en re-edición / creación ——— OBJETIVO Ajustar GetInitialContentAsync (caso de uso “Initial Content”) para que la hidratación de plantillas no dependa de tokens detectados en el HTML como “todos requeridos”, sino que use el catálogo oficial ra_editor_template_tokens: validar requeridos con is_required = 1 permitir tokens opcionales / editables con default_value resolver valores por source_type: respuesta_api system static editable Con esto se elimina el error actual: “Faltan tokens requeridos para hidratar la plantilla” cuando faltan tokens como Fecha, Ciudad, TextoRespuesta (que en BD están definidos como system/static/editable, no como respuesta_api). ——— CONTEXTO / PROBLEMA Hoy GetInitialContentAsync: extrae tokens desde template_html (regex {{Token}}) trata todos los tokens encontrados como requeridos construye mapa de tokens solo con datos provenientes de la “respuesta API” (ej. RaRespuestaRadicado u objeto similar) falla si algún token no está presente o viene vacío Pero el diseño correcto ya existe en BD: Tabla: ra_editor_template_tokens token_code source_type (respuesta_api, system, static, editable) source_key is_required default_value Ejemplo tokens plantilla: Fecha → system / FechaActual Ciudad → static / default_value TextoRespuesta → editable / default_value ——— ALCANCE (REQUERIDO) 1) Usar tokens oficiales de BD Para una plantilla seleccionada (template_definition_id): consultar ra_editor_template_tokens por template_definition_id construir lista “tokens esperados” desde BD (no desde regex) validar requeridos solo con is_required=1 “La selección de la plantilla debe seguir el mecanismo oficial configurado en BD (si existe ra_editor_template_context_rules): resolver template_definition_id por context_definition_id derivado de ContextCode.” Nota: Se puede seguir usando regex solo para reemplazo en HTML, pero la “lista oficial” de tokens viene de BD. 2) Resolver valores por source_type Implementar resolver: respuesta_api: tomar propiedad/campo desde el objeto DTO de negocio usando source_key (reflection) o un diccionario controlado. system: FechaActual → fecha actual en formato definido (definir convención: yyyy-MM-dd o el estándar existente del proyecto). otros system deben estar soportados por catálogo (switch controlado). static: usar default_value. editable: usar default_value (puede ser vacío), y no fallar salvo que is_required=1 y no exista valor final. 3) Manejo de errores consistente Casos: no existe plantilla o versión activa → error controlado no existen tokens configurados para la plantilla → error controlado (o permitir 0 si el HTML no tiene tokens; definir criterio) faltan tokens requeridos (según is_required) → error controlado con lista de tokens faltantes excepciones → AppResponses con Type "Exception" 4) Persistir/retornar trazabilidad de tokens (si aplica) Si el endpoint actual retorna tokens resueltos (o se requiere): retornar TokensResueltos (token_code → valor) marcar si provino de default_value vs respuesta_api (opcional, si ya existe estructura) ——— NO ALCANCE No cambiar reglas de negocio del módulo (solo hidratación técnica). No modificar SolicitaEstructuraRespuestaIdTareaAsync (tiene ticket aparte). No crear nuevo endpoint (se mantiene el existente de initial content).
